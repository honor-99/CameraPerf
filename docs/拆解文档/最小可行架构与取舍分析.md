# CameraPerf 最小可行架构 & 技术取舍分析

> 目标：基于 CameraPerf 核心，裁剪出 **Trace 智能拆解工具 + Log 分析工具**
> 原则：Android 分析领域知识（skills/strategies）是核心资产，应保留；平台工程（agentv2/selfImprove/多模型路由等）可剥离

---

## 一、核心资产 vs 平台包袱 — 重新分类

### 核心资产（保留，是你要继续深耕的）

| 资产 | 内容 | 价值 | 需要做的事情 |
|------|------|------|-------------|
| **Skill 引擎** (skillEngine/) | 通用 YAML DSL 执行引擎 | 分析流水线的运行时 | 直接复用 |
| **Skill 资产库** (skills/) | 150+ YAML，Android trace 分析知识 | ⭐ 最核心的领域资产 | 保留 → 按新需求增删 |
| **策略模板库** (strategies/) | 40+ Markdown 策略，场景分析 SOP | ⭐ 最核心的决策逻辑 | 保留 → 按新需求增删 |
| **trace_processor_shell 通信** | Perfetto SQL 引擎管理 | trace 查询的基础设施 | 直接复用 |
| **Claude Agent SDK 编排** | agentv3 核心运行时 | AI 驱动的分析决策 | 精简保留 |
| **MCP 工具框架** | 20 个工具，连接 Agent↔Skill↔SQL | Agent 能力边界 | 裁剪到 8-10 个核心工具 |
| **PerfettoSQL 知识库** | stdlib 表/函数索引 | SQL 查询正确性的保障 | 直接复用 |
| **DataEnvelope 数据契约** | 前后端统一数据格式 | 结果展示的基础 | 直接复用 |

### 平台包袱（剥离，跟分析 trace/log 无关）

| 模块 | 行数估算 | 为什么是包袱 |
|------|---------|-------------|
| **agentv2** | ~30K | Legacy DeepSeek 降级路径，已被 agentv3 完全取代 |
| **selfImprove/** | ~8K | 自动修补策略、审查队列、反馈管道——过度工程，初期完全不需要 |
| **多 Provider 切换** | ~3K | Gemini/GLM/DeepSeek/Qwen/Kimi/Doubao… 初期单模型足够 |
| **场景重建** (sceneReconstruction) | ~5K | Perfetto UI 选区→自动分段分析，过于复杂 |
| **Teaching 教学系统** | ~3K | 渐进式分析教学链路，独立场景 |
| **CI Gate** | ~2K | CI/CD 集成，跟分析工具无关 |
| **RAG Admin** | ~1K | 独立的 RAG 管理界面 |
| **Flamegraph Analyzer** (Rust) | ~3K | 火焰图分析，可选独立模块 |
| **HTML 报告生成** | ~2K | 可简化为 Markdown 输出 |
| **adbTools / traceRecorder** | ~5K | 抓 trace 的客户端工具，独立于分析引擎 |
| **多轮对话持久化** | ~5K | sessionPersistence/sessionLogger/checkpointManager，初期单轮够用 |

## 六、关键架构决策

### 决策 1: Skill 引擎要不要动？

**不动。** 它是唯一可以同时驱动 Trace 和 Log 两个数据源的执行引擎。9 种步骤类型（atomic/composite/iterator/parallel/conditional/diagnostic/ai_decision/ai_summary/pipeline）已经足够通用。

### 决策 2: trace_processor_shell vs 自建 SQLite？

**保留 trace_processor_shell 用于 Trace 分析。** Perfetto 的 stdlib（android.frames.timeline 等）是 trace 分析的基础，不可能自己重写。

**Log 分析用独立 SQLite。** Log 数据的查询模式（文本搜索、时间范围、模式匹配）与 trace（时序事件 JOIN）不同，用独立数据库更合适。

### 决策 3: 要保留 Claude Agent SDK 绑定吗？

**保留。** 初期不需要支持多模型。但应把 `claudeConfig.ts` 设计为可注入的适配器接口，方便以后接入其他模型。

### 决策 4: 策略模板要简化吗？

**内容不简化，加载方式简化。** 策略模板里的 Android 分析知识是核心资产，不能删。但可以简化策略加载逻辑（去掉 sceneReconstruction 的复杂路由，保留基本的场景匹配+模板注入）。

### 决策 5: 前后端分离还是单体？

**初期单体。** Perfetto UI 是沉重的依赖（需要 submodule + C++ 编译）。第二阶段可以做一个轻量 Web UI，只做上传 + 提问 + 看结果。CLI 优先。
