AI 辅助开发流程的 4 个阶段

人的介入从执行层逐步上移到决策层

01
直接对话
描述需求 → AI 直接改代码
类似结对编程：一个人说，一个人写。人需要逐行审查每次修改，因为 AI 对项目上下文理解有限，经常做出不符合整体架构的局部修改。
我: "把这个函数改成支持批量处理"
AI: [直接修改代码]
我: [逐行 review 每次改动]
Review 重心
逐行代码
人工介入: 高
▼
02
Plan Mode
需求 → 实施方案 → 确认 → 执行
AI 先输出结构化方案（改哪些文件、改什么、改动顺序），人审查方案后再执行。Review 重心从逐行代码转移到架构方案。
我: "添加架构检测功能"
AI: [输出方案: 文件列表 + 变更摘要 + 依赖关系]
我: [审查方案, 确认]
AI: [按方案执行]
Review 重心
架构方案
人工介入: 中
▼
03
+ Codex Review
方案 → 独立 Review → 修正 → 执行
引入 Codex 作为独立 reviewer，以只读模式访问代码库，从架构合理性、边界情况、遗漏风险三个角度审查方案。相当于 AI 开发流程中的 code review 环节。
AI (Claude): [输出方案]
AI (Codex): [只读 review: 架构 + 边界 + 遗漏]
我: [评估反馈, ~20% 驳回并说明理由]
AI (Claude): [按修正方案执行]
Review 重心
Codex 反馈评估
人工介入: 中低
▼
04
完整流水线
需求 → Plan → Review → Execute → Test → Simplify → Final Review
人只介入需求定义和 review 反馈评估。代码执行、测试（tsc + 6 trace 回归 + E2E Agent 分析）、代码整理由工程流水线完成。
我: [确定需求和架构方向]
Claude: Plan → Codex: Review → 我: 评估
Claude: 执行 → tsc + regression + E2E
→ /simplify → Codex: Final Review
Review 重心
需求 + 架构决策
人工介入: 低
Harness Engineering 的演进方向：构建足够的工程基础设施，使人可以信任 AI 的执行结果，把注意力放在更高层的决策上
